# 数据管理页面 — 规格说明

> 依据仓库内 `.plan/数据管理页面.plan.md` 整理；本文档描述**产品与接口级规格**（与实现保持同步）。  
> 与《导入路网数据》《导入路段数据》规格中的既有接口保持对齐。

---

## 1. 目标与范围

在**后台管理**（与现有 `AgentPageShell` 侧栏同一套壳）中提供「数据管理」能力：

1. **项目列表**：以「项目」为数据域容器，用户先维护项目，再在项目下做导入与查看记录。  
2. **列表能力**：分页（默认每页 20 条）、按**项目名称**模糊查询、手动刷新。  
3. **操作**：每行提供「**导入**」入口（进入**该项目专属**的导入子页）、「**清除**」入口（见 **§3.4**），以及「**删除**」入口（见 **§3.5**）。  
4. **导入子页**：包含四个区块——**导入记录**、**导入路网数据**、**导入路段数据**、**导入病害数据**；各导入 Tab 内「导入」按钮旁提供**同类数据清除**（见 **§4.4**）。  
5. **导入记录**：每次通过本功能触发的导入（无论成功失败）写入一条记录，至少包含：**导入时间**、**导入用时**、类型、文件名、状态、失败原因（若有）、结果摘要（如条数等）。

| 路由（建议） | 说明 |
|--------------|------|
| `/admin/data-management` | 项目列表（表格 + 查询 + 刷新 + 新建项目） |

## 3. 项目列表（首页）

### 3.1 字段（列表展示）

| 列 | 说明 |
|----|------|
| 项目名称 | 主展示字段 |
| 备注 | 可选 |
| 创建时间 / 更新时间 | 便于排序与审计 |
| 操作 | 「**导入**」跳转导入子页；「**清除**」执行 **§3.4**（保留项目主档）；「**删除**」执行 **§3.5**（先等同 **§3.4** 全量清除，再软删项目主档）；可选「编辑」未在本期强制要求 |

### 3.2 查询与分页

- **分页**：`pageNo` 从 1 开始，`pageSize` 默认 **20**，可改 10/20/50。  
- **模糊查询**：`nameKeyword` 对项目名称 `LIKE`（前后 `%`，注意索引与性能，数据量大时可改为前缀匹配或独立搜索表）。  
- **刷新**：重新请求当前页，不改变查询条件。

### 3.3 新建项目

- 表单：**项目名称**（必填）、**备注**（选填）。  
- 校验：名称非空、长度上限与库表一致（建议 200 字以内）。

### 3.4 列表「清除」（全项目数据域）

- **语义**：在**当前租户**下，对 `project_id = {projectId}` 且 `deleted = false` 的 **`road_route` / `road_section` / `disease_record`** 行执行**软删**（将 `deleted` 置为 `true`），并**软删**本项目下 **`data_import_record`** 流水（便于重新导入后记录表从空开始展示）。  
- **不动**：**`data_mgmt_project` 项目主档**本身不删除；**`project_id` 为 NULL** 的历史 / GIS 直连数据不受影响。  
- **交互**：操作前必须 **二次确认**（文案明确列出将影响的范围）；成功后提示各表**实际软删行数**（可为 0）。  
- **接口**：见 **§4.5**，`scope` 取 **`ALL`**。

### 3.5 列表「删除项目」

- **语义**：与 **§3.4** 一致先软删本项目归属的 **`road_route` / `road_section` / `disease_record`** 及 **`data_import_record`**，再对 **`data_mgmt_project`** 行执行**软删**（`deleted = true`）。须在**同一事务**中完成，任一步失败则整体回滚。  
- **与「清除」的差异**：**清除**不删项目主档，列表仍可见该项目；**删除**后项目从列表消失（查询过滤 `deleted = false`），但库中主档与业务行均为软删，可审计恢复。  
- **交互**：二次确认，文案须明确「先清数据再删项目」及不可从列表恢复的含义。  
- **接口**：见 **§4.6**。已删除项目的 `id` 不应再用于导入子页；若用户持有旧链接访问，应提示项目不存在或已删除。

## 4. 导入子页

### 4.1 Tab 一：导入记录

- **数据源**：本项目下全部导入流水，按**开始时间倒序**。  
- **分页**：与列表一致，默认 20。  
- **建议列**：开始时间、结束时间、耗时(ms)、导入类型、文件名、状态（成功/失败）、简要信息（成功时可展示 JSON 摘要中的关键数字；失败时展示 `message` 截断）。

### 4.2 Tab 二～四：三种导入

前端 Tab 调用 **§4.3 编排接口**（带 `projectId`）；编排层内部再调用**既有导入 Service**（与直接调 `POST /api/road-routes/import-network` 等**业务语义一致**），避免重复解析与事务规则。

| Tab | 编排路径（§4.3） | 内部等价既有能力（参考） |
|-----|-------------------|---------------------------|
| 导入路网 | `.../imports/road-network` | 原 `POST /api/road-routes/import-network` |
| 导入路段 | `.../imports/section-package` | 原 `POST /api/road-sections/import-package` |
| 导入病害 | `.../imports/disease-excel` | 原 `POST /api/road-assets/diseases/import-excel` |

请求仍为 `multipart/form-data`、字段名 `file`；前端使用 **`longRequest`**（大超时），`X-Tenant-Id` 与现网一致。

### 4.3 与「项目」的关联方式 **【已定】方案 A**

采用 **编排接口**：由后端在**一次请求**内完成「校验项目 → 调用既有导入 Service → 写入导入流水」，路径形如（实现时可微调前缀，但需全局统一）：

| 能力 | 建议路径 |
|------|----------|
| 路网 | `POST /api/data-mgmt/projects/{projectId}/imports/road-network` |
| 路段包 | `POST /api/data-mgmt/projects/{projectId}/imports/section-package` |
| 病害 Excel | `POST /api/data-mgmt/projects/{projectId}/imports/disease-excel` |

- 编排层内部转调现有 `RoadNetworkImportService`、`RoadSectionPackageImportService`、`DiseaseExcelImportService`（或当前仓库中等价实现），**不复制**解压 / Shapefile / Excel 解析逻辑。  
- **业务表冗余 `project_id`**：编排导入在**成功落库**后，须将本批受影响的路网 / 路段 / 病害行写入 **`project_id`**（与路径参数 `{projectId}` 一致），便于按项目做 GIS 过滤、统计与导出；具体落点见 **§5.0**（必要时在既有 Service 内增加「由编排层传入的 projectId」参数，或在编排层对 Mapper 做批量 `UPDATE`，实现 PR 中选定一种并说明）。  
- **导入流水**：每次调用无论成功或失败，均写入 `data_import_record`（见 §5）；**事务边界**在实现 PR 中写明（例如：业务导入已独立 `@Transactional` 时，流水是否单独提交以保证失败可追溯——推荐**流水与调用结果一致可查**，失败也落库）。

**方案 B**（前端分两次调接口补记流水）**不采用**。

### 4.4 各 Tab「清除」（按数据类型）

- **路网 Tab**：清除仅针对 **`road_route.project_id = {projectId}`** 的未删除行（软删）；**不**删路段、病害、导入流水。  
- **路段 Tab**：清除仅针对 **`road_section.project_id = {projectId}`** 的未删除行；**不**删路线、病害、导入流水。  
- **病害 Tab**：清除仅针对 **`disease_record.project_id = {projectId}`** 的未删除行；**不**删路线、路段、导入流水。  
- **导入记录 Tab**：不提供清除按钮（避免与列表「全量清除」语义混淆）；若需清空流水，使用列表 **§3.4** 或后续单独「仅清流水」能力（本期未单独暴露）。

各 Tab 清除前须 **二次确认**；`scope` 与 **§4.5** 中枚举一致：`ROAD_NETWORK` / `SECTION_PACKAGE` / `DISEASE_EXCEL`。

### 4.5 清除编排接口

| 能力 | 路径 | 方法 | 请求体 |
|------|------|------|--------|
| 按项目清除 | `POST /api/data-mgmt/projects/{projectId}/clear` | JSON | `{ "scope": "<枚举>" }` |

**`scope` 取值**：

| 值 | 行为 |
|----|------|
| `ALL` | 软删本项目下路网、路段、病害中已带 `project_id` 的行 **以及** 本项目 **`data_import_record`** 流水（均未删除的）。执行顺序建议：病害 → 路段 → 路线 → 流水，避免残留引用语义混乱（实现以事务内批量 `UPDATE` 为准）。 |
| `ROAD_NETWORK` | 仅软删 **`road_route`**（`tenant_id` + `project_id` 匹配）。 |
| `SECTION_PACKAGE` | 仅软删 **`road_section`**。 |
| `DISEASE_EXCEL` | 仅软删 **`disease_record`**。 |

**响应体建议**：返回各表本次 `UPDATE` 影响行数（`routes`、`sections`、`diseaseRecords`、`importRecords`），供前端 Toast 展示。

**安全**：必须校验项目存在且归属当前租户；所有 `UPDATE` 须带 `tenant_id` 条件。

### 4.6 删除项目接口

| 能力 | 路径 | 方法 | 请求体 |
|------|------|------|--------|
| 删除项目 | `DELETE /api/data-mgmt/projects/{id}` | 无 | 无 |

- **行为**：校验项目存在且未删除 → 执行与 **§4.5** `scope=ALL` **相同的数据软删逻辑** → 软删 **`data_mgmt_project`**（`tenant_id` + `id` 匹配且 `deleted=false`）。  
- **响应**：`R` 成功即可；可不返回体。  
- **实现注意**：与 **§4.5** 复用同一套清除实现，避免两套 SQL 分叉；注意 **Service 循环依赖**（可用 `@Lazy` 或独立编排 Bean 拆解）。

## 5. 数据库设计建议（实现阶段补充 DDL）

### 5.0 项目与关联建模原则 **【已定】**

采用 **「项目主档 + 业务表冗余 `project_id` + 导入流水」** 三层结构（产品语义：既能在库里按项目筛数据，又保留每次上传的审计记录）。

#### （1）项目主档（仍是一张表）

- **`data_mgmt_project`**：仅存项目的**主数据**（名称、备注、租户、审计字段等），作为 `project_id` 的**语义来源**与后台「项目列表」的数据源。  
- 扩展项目属性（业主、合同号等）：**优先加在本表**，避免项目维度拆成多主档表。

#### （2）业务表冗余 `project_id` **【已定】**

在下列表中增加 **`project_id` `VARCHAR(64) NULL`**（列名与 Java 字段 `projectId` 对应；库列建议 `project_id`）：

| 表 | 说明 |
|----|------|
| **`road_route`** | 路网路线；通过**数据管理—编排导入**写入或更新的行，须带上当前项目的 `project_id`。 |
| **`road_section`** | 路段；同上。 |
| **`disease_record`** | 病害记录；同上。 |

约定与边界：

- **`NULL` 语义**：历史数据或 **GIS 一张图直接调用旧有导入接口**（不带 `projectId`）产生的行，允许 **`project_id` 为空**，表示「未归属任一数据管理项目」。  
- **非空语义**：仅当数据来自 **`/api/data-mgmt/projects/{projectId}/imports/...`** 且导入**成功提交**后，对应新建或更新的业务行应 **`project_id = {projectId}`**（与 `data_mgmt_project.id` 一致）。  
- **更新类导入**（如路网按 `tenant_id + route_code` upsert）：若同一 `route_code` 再次被**另一项目**编排导入，**【已定】** 将 `project_id` **更新为本次请求的项目**（以编排入口为准；若产品后续要求禁止跨项目覆盖，再单开规则）。  
- **`road_evaluation_unit` / `assessment_result` 等**：本期**不要求**强制加 `project_id`；若需按项目过滤评定单元，可通过 `road_section.project_id` 或 `route_id` 关联推导。若后续产品要求单元/结果也直查项目，再扩展 DDL。

#### （3）导入流水表（不变）

- **`data_import_record.project_id`**：仍指向 **`data_mgmt_project.id`**，记录「**是哪次、在哪个项目下**触发的上传」，与业务行上的冗余 **`project_id`** 并存：前者偏**审计与次数**，后者偏**业务查询与权限过滤**。  
- **不把项目主档合并进 `data_import_record`**：流水表只存事件与结果摘要。

### 5.1 `data_mgmt_project`（项目）

| 列 | 类型建议 | 说明 |
|----|----------|------|
| id | varchar(64) PK | UUID |
| tenant_id | varchar(64) | 与全库租户一致 |
| name | varchar(200) | 项目名称 |
| remark | text | 可选 |
| created_by / updated_by / created_at / updated_at / deleted | 与现有 `BaseEntity` 风格一致 |

索引：`(tenant_id, name)` 可选（若需唯一项目名则加唯一约束，产品未强制）。

### 5.2 `data_import_record`（导入记录）

| 列 | 类型建议 | 说明 |
|----|----------|------|
| id | varchar(64) PK | |
| tenant_id | varchar(64) | |
| project_id | varchar(64) | **必填**；关联 **`data_mgmt_project.id`**（与 §5.0（3）一致） |
| import_type | varchar(32) | 枚举：`ROAD_NETWORK` / `SECTION_PACKAGE` / `DISEASE_EXCEL` |
| file_name | varchar(500) | 原始文件名 |
| started_at | timestamp | 开始 |
| finished_at | timestamp | 结束 |
| duration_ms | bigint | 用时 |
| status | varchar(20) | `SUCCESS` / `FAILED` |
| message | text | 失败原因或补充说明 |
| result_summary | text | 成功时建议存各接口返回 VO 的 **JSON 子集**（条数、warnings 条数等） |

索引：`(tenant_id, project_id, started_at DESC)` 便于列表页。

### 5.3 业务表 DDL 与实体 **【待实现清单】**

- 为 **`road_route` / `road_section` / `disease_record`** 增加 `project_id` 列及索引（建议 `(tenant_id, project_id)` 或 `(tenant_id, project_id, deleted)`，与现有查询习惯对齐）。  
- 同步 **`RoadRoute` / `RoadSection` / `disease` 模块下病害实体**（以仓库实际类名为准）及 **Mapper / `insert`·`update`·GIS 查询 SQL`**：凡编排导入写库路径须能写入或刷新 `project_id`；GIS 列表接口若需「按当前项目过滤」，增加可选查询参数 `projectId`。

## 6. 后端模块归属建议

- **表 + 实体 + Mapper + 项目 CRUD + 导入流水查询**：可放在 **`srmp-road-asset`**（与路网/路段/病害导入同模块），或单独小模块；**避免**在 `srmp-admin` 写业务 SQL。  
- **编排导入 + 写流水 + 业务表 `project_id`**：按 **§4.3 方案 A** 与 **§5.0（2）**，与既有导入 Service 组合，并扩展写库与查询以支持冗余 **`project_id`**。  
- **按项目清除（软删）**：与编排同模块实现 **`POST .../clear`**（§4.5），对 `RoadRoute` / `RoadSection` / `DiseaseRecord` / `DataImportRecord` 使用带 `tenant_id` + `project_id` 条件的批量 `UPDATE`（与 **§3.4、§4.4** 一致）。  
- **删除项目**：实现 **`DELETE .../projects/{id}`**（§4.6），编排层先复用 **§4.5** 的 `ALL` 清除，再软删 **`data_mgmt_project`**。

## 7. 前端要点

- 列表页与导入页均使用 **`AgentPageShell`**，保持与现有后台一致。  
- 导入 Tab 内 **不要** 手写 `Content-Type: multipart/form-data` 的 boundary；使用 `FormData` + axios 默认行为。  
- 成功/失败提示：沿用全局 `ElMessage` + 后端 `R.message`；记录 Tab 在导入成功后**自动刷新当前页**。  
- **清除**：列表「清除」与导入 Tab「清除 *」均使用 **`ElMessageBox.confirm`** 二次确认；清除成功后可用接口返回的各表影响行数组装 Toast（见 **§4.5**）。  
- **删除项目**：列表「删除」须 **`ElMessageBox.confirm`**，明确与「清除」的差异；成功后刷新分页列表（见 **§3.5、§4.6**）。
