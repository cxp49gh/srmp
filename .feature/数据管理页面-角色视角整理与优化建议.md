# 数据管理页面 — 角色视角整理与优化建议

> 生成日期：2026-05-19  
> 范围：数据管理父子菜单、`/admin/data-management` 项目列表页、`/admin/data-management/:projectId/import` 项目导入页，以及与 GIS 一张图项目筛选的联动。  
> 依据：当前前端页面、`data-mgmt` 后端接口、`.feature/数据管理页面-规格说明.md`、路网/路段/病害导入规格。

---

## 1. 当前页面定位

数据管理页面现在承担三个核心职责：

1. 以“项目”为数据域，统一组织路网、路段、病害等导入数据。
2. 在项目下执行路网 `.tar`、路段包 `.tar`、病害 `.xlsx` 上传，并记录导入流水。
3. 支撑 GIS 一张图按 `projectId` 加载对应项目数据，避免不同批次、不同项目数据混用。

当前能力已经具备基本闭环：项目创建、项目列表、按项目导入、导入记录、按类型清除、全量清除、删除项目、跳转 GIS 一张图。

但页面仍偏“工程入口”，对不同角色的任务、风险和判断信息补充不足。后续优化目标应从“能上传”升级为“能管理、能核查、能追溯、能协同”。

同时，当前能力集中在单一“数据管理”入口下，功能层次不够清晰。建议将“数据管理”改为可折叠父菜单，并按功能拆成多个子菜单，让用户按任务进入对应页面。

---

## 2. 角色与核心诉求

| 角色 | 典型目标 | 当前满足度 | 主要缺口 |
|------|----------|------------|----------|
| 普通业务用户 | 找到正确项目，查看数据是否可用，进入 GIS 分析 | 中 | 缺项目数据概览、导入状态解释、错误修复指引 |
| 管理员 | 创建项目、导入/清除/删除数据、控制风险 | 中 | 缺权限分级、危险操作审计、删除口径统一 |
| 数据人员 | 准备文件、导入、排查失败、确认入库质量 | 低-中 | 缺模板/格式说明、预校验、失败明细、质量报告 |
| 运维人员 | 定位接口异常、数据残留、租户隔离和性能问题 | 低 | 缺 traceId、任务日志、耗时分段、异常上下文 |
| 领导/查看人员 | 快速判断项目数据是否完整、是否可用于汇报 | 低 | 缺项目状态、数据覆盖率、最新更新时间和成果入口 |
| 审计/安全人员 | 追踪谁在什么时候导入、清除、删除了什么 | 低 | 缺操作人、审批/确认记录、删除前后影响范围快照 |

## 3. 普通业务用户视角

### 3.1 需要补充的内容

| 页面区域 | 应补充内容 | 说明 |
|----------|------------|------|
| 项目列表 | 项目状态 | 建议展示 `未导入`、`部分导入`、`可用`、`导入失败`、`已清空`。用户不应只靠创建时间判断项目是否可用。 |
| 项目列表 | 数据概览 | 展示路线数、路段数、病害数、最后导入时间、最后导入状态。 |
| 项目列表 | 主要入口 | “一张图”入口应根据项目数据状态给出提示：无数据时提示先导入；有失败记录时提示先核查。 |
| 导入页 | 当前项目摘要 | 顶部展示项目名称、备注、数据状态、最近一次成功导入时间。 |
| 导入记录 | 人可读结果 | `resultSummary` 当前容易是 JSON 文本，建议解析成“新增/更新/跳过/警告”指标。 |
| 空状态 | 下一步动作 | 无项目、无记录、无数据时，应给出明确按钮：新建项目、导入路网、查看格式要求。 |
| 导入顺序 | 路网前置提示 | 路段、病害依赖路网，当前项目没有路网时应禁用上传并提示先导入路网。 |

### 3.2 交互优化

- 项目名称搜索应支持回车触发查询，并在重置后回到第一页。
- 项目列表建议增加“最近更新倒序 / 创建时间倒序 / 名称排序”。
- “导入”“一张图”“清除”“删除”视觉权重应区分：常用操作前置，危险操作收纳或弱化。
- 进入 GIS 一张图时，应把项目名带入页面显著位置，避免用户不知道当前看的是哪个项目。
- GIS 中项目下拉为空时，应提示“暂无数据管理项目，请先到数据管理新建项目”，并提供跳转。

### 3.3 用户验收标准

1. 用户不用打开导入记录，也能在项目列表判断项目是否已有可用数据。
2. 用户从项目列表进入一张图后，地图能明确显示当前项目名称和筛选状态。
3. 导入失败时，用户能看到失败原因、影响文件和下一步修复建议。

## 4. 管理员视角

### 4.1 需要补充的内容

| 能力 | 建议补充 | 原因 |
|------|----------|------|
| 项目管理 | 编辑项目名称、备注、负责人、所属年度、数据来源单位 | 当前只能新建，后期项目主数据维护不足。 |
| 权限控制 | 区分查看、导入、清除、删除、项目维护权限 | 清除/删除是高风险操作，不应所有后台用户都可执行。 |
| 危险操作 | 删除/清除前展示影响范围预估 | 二次确认文案不足以支撑管理员判断风险。 |
| 操作审计 | 记录操作人、操作时间、操作类型、操作前数据量、操作后结果 | 便于追责、恢复和问题定位。 |
| 生命周期 | 项目状态增加“草稿、导入中、可用、归档、已删除” | 避免删除项目承担“归档”语义。 |

### 4.2 当前不足点

1. “清除”和“删除”目前页面文案写明会物理删除业务数据，但旧规格中仍有“软删业务数据”的描述。产品、文案和后端实现需统一口径。
2. 删除项目当前是“先物理删除业务数据与导入流水，再软删项目主档”。这对审计和恢复不友好，应至少补充删除前影响快照或导出能力。
3. 清除单类数据时，可能破坏项目数据完整性。例如只清路网但保留路段/病害，会让 GIS 关联和统计解释变复杂。页面需要展示依赖风险。
4. 当前没有项目归档能力。对于不再使用但需要保留历史的项目，管理员只能保留在列表或删除，缺少更合适的生命周期操作。
5. 路段、病害应以项目路网为前置条件。管理员清除路网后，应自动阻断后续路段/病害导入，直到该项目重新导入路网。

### 4.3 管理员优化建议

| 优先级 | 建议 | 说明 |
|--------|------|------|
| P0 | 统一删除口径 | 明确业务数据是物理删除还是软删；文档、按钮文案、接口注释保持一致。 |
| P0 | 危险操作影响预览 | 清除/删除前调用预览接口返回路线、路段、病害、导入流水数量。 |
| P0 | 路网前置校验 | 路段/病害导入前必须校验当前项目已有路网，前端禁用入口，后端强校验。 |
| P1 | 权限分级 | 至少拆分 `data:view`、`data:import`、`data:clear`、`data:delete`、`data:project:edit`。 |
| P1 | 项目编辑 | 支持修改项目名称、备注、负责人、年度、来源单位。 |
| P1 | 归档能力 | 归档项目从默认列表隐藏，但数据仍可查可审计。 |
| P2 | 删除审批 | 生产环境可选开启删除审批或要求输入项目名称确认。 |

## 5. 数据人员视角

### 5.1 需要补充的内容

| 阶段 | 当前体验 | 建议补充 |
|------|----------|----------|
| 导入前 | 只看到简短格式提示 | 提供模板下载、示例包说明、字段要求、坐标系要求、文件大小限制。 |
| 选择文件 | 直接上传 | 增加文件名、大小、类型校验；显示本次会导入到哪个项目。 |
| 导入中 | 按钮 loading | 大文件导入建议有任务进度或至少展示“解析中/入库中/写流水中”。 |
| 导入后 | 成功 toast + 记录刷新 | 展示导入结果面板：新增、更新、跳过、警告、失败明细。 |
| 失败排查 | 只有 message | 支持下载错误报告，包含文件、sheet、行号、字段、原因、建议修正方式。 |
| 质量确认 | 需去 GIS 或数据库判断 | 导入后给出质量检查：坐标范围、路线匹配率、病害缺路线数、空几何数等。 |

### 5.2 三类导入的专项补强

| 类型 | 缺少内容 | 优化建议 |
|------|----------|----------|
| 路网 | 缺少 Shapefile 包结构说明 | 展示 `.tar` 内必须包含 `.shp/.dbf/.shx/.prj`，且只能有一组路网 Shapefile。 |
| 路段 | 四级路段复杂，页面提示过少 | 明确线路级、台账级、公里级、百米级命名规则和缺失任一级的处理规则。 |
| 病害 | Excel 表头规则严格但页面未说明 | 展示“第 5 行表头、第 6 行数据”要求，并提供标准模板下载。 |

### 5.3 导入顺序规则

路网是项目数据的空间骨架，路段和病害都需要依赖路网路线编号、路线几何和项目归属关系。因此数据管理页应明确采用以下顺序：

1. 先导入路网数据。
2. 路网导入成功后，才允许导入路段数据。
3. 路网导入成功后，才允许导入病害数据。

页面表现建议：

| 场景 | 页面处理 |
|------|----------|
| 新建项目但未导入路网 | 路段、病害上传按钮置灰，提示“请先导入路网数据”。 |
| 路网导入失败 | 仍视为未具备路网，禁止导入路段和病害；导入记录显示失败原因。 |
| 路网导入成功且路线数大于 0 | 开放路段、病害上传入口。 |
| 清除路网数据后 | 重新禁用路段、病害上传入口，并提示需重新导入路网。 |
| 项目已有路段/病害但路网被清除 | 页面给出数据依赖风险提示，避免用户误判项目仍完整可用。 |

校验建议：

- 前端根据项目数据概览判断是否开放路段、病害上传。
- 后端导入接口也应做同样校验，避免绕过页面直接上传。
- 校验口径以“当前项目下已归属、未删除的路网路线数大于 0”为准。
- 错误提示统一为“请先导入路网数据，再导入路段或病害数据”。

### 5.4 数据质量指标

导入成功后建议在记录详情中展示：

- 路网：新增路线数、更新路线数、跳过要素数、坐标系、警告数。
- 路段：线路级/台账级/公里级/百米级入库数、评定结果入库数、未匹配路线数。
- 病害：导入病害数、未匹配路线数、未分类病害数、经纬度异常数、图片地址缺失数。
- 通用：文件名、文件大小、上传人、开始时间、结束时间、耗时、租户、项目。

## 6. 运维与审计视角

### 6.1 运维需要补充的内容

| 信息 | 用途 |
|------|------|
| traceId / requestId | 前端报错、后端日志、网关日志关联。 |
| 接口耗时分段 | 区分上传慢、解压慢、解析慢、入库慢、GIS 查询慢。 |
| 文件元信息 | 文件名、大小、hash、上传客户端、租户。 |
| 异常类型 | 业务校验失败、解析失败、数据库失败、超时、权限失败。 |
| 重试建议 | 哪些失败可重试，哪些必须修文件。 |

### 6.2 审计需要补充的内容

| 操作 | 审计字段 |
|------|----------|
| 新建项目 | 操作人、时间、项目名称、备注、租户。 |
| 编辑项目 | 修改前后字段。 |
| 导入数据 | 上传人、文件 hash、导入类型、结果摘要、失败原因。 |
| 清除数据 | 操作人、清除范围、清除前数量、清除后数量、确认方式。 |
| 删除项目 | 操作人、项目快照、删除前数据量、删除原因。 |

建议新增独立 `data_mgmt_operation_log` 或复用系统审计日志，避免把管理操作全部塞进 `data_import_record`。

## 7. 菜单与页面结构优化方案

### 7.1 父子菜单拆分

当前“数据管理”承载项目、导入、记录、质量、模板、审计等多个功能，单页承载会让普通用户和管理员都难以快速定位任务。建议改为父子菜单结构：父级菜单只负责聚合和折叠，子菜单按功能进入不同页面。

父级菜单：

| 父级菜单 | 展示方式 | 默认状态 |
|----------|----------|----------|
| 数据管理 | 左侧导航一级菜单，可展开/折叠 | 默认展开；用户折叠后保留折叠状态 |

子菜单建议：

| 子菜单 | 路由建议 | 主要用户 | 核心功能 |
|--------|----------|----------|----------|
| 项目总览 | `/admin/data-management/projects` | 普通业务用户、领导、管理员 | 项目列表、项目状态、数据概览、进入 GIS、进入项目详情。 |
| 项目导入 | `/admin/data-management/projects/:projectId/import` | 数据人员、管理员 | 路网、路段、病害导入；路网前置校验；导入结果反馈。 |
| 导入记录 | `/admin/data-management/import-records` | 数据人员、运维、审计 | 跨项目查看导入流水，按项目、类型、状态、时间筛选。 |
| 数据质量 | `/admin/data-management/quality` | 数据人员、领导、管理员 | 项目质量报告、路线匹配率、未分类病害、空几何、异常坐标。 |
| 模板与规范 | `/admin/data-management/templates` | 数据人员 | 下载路网、路段、病害模板；查看文件格式、字段、坐标系、示例包说明。 |
| 清除与归档 | `/admin/data-management/maintenance` | 管理员 | 清除预览、清除执行、项目归档、恢复、删除前影响确认。 |
| 操作审计 | `/admin/data-management/audit` | 管理员、审计、安全人员 | 查看项目创建、导入、清除、删除、归档等操作日志。 |

### 7.2 折叠菜单交互

| 场景 | 交互规则 |
|------|----------|
| 首次进入后台 | “数据管理”父菜单默认展开，并高亮当前子菜单。 |
| 用户点击父菜单 | 展开或折叠子菜单，不直接跳转；若必须跳转，则跳转到“项目总览”。 |
| 用户进入任一子页面 | 父菜单保持展开，当前子菜单高亮。 |
| 左侧导航收起为窄栏 | 父级只显示图标；悬浮或点击后浮层展示子菜单。 |
| 权限不足 | 隐藏无权限子菜单；父菜单只在至少一个子菜单可见时展示。 |
| 移动端或窄屏 | 父菜单折叠为抽屉导航，点击子菜单后自动收起。 |

菜单状态建议保存在前端本地状态中，避免用户切换页面后父菜单反复自动折叠。若系统已有统一侧栏状态管理，应复用现有实现，不为数据管理单独发明一套导航状态。

权限建议：

| 子菜单 | 权限建议 |
|--------|----------|
| 项目总览 | `data:project:view` |
| 项目导入 | `data:import` |
| 导入记录 | `data:import-record:view` |
| 数据质量 | `data:quality:view` |
| 模板与规范 | `data:template:view` |
| 清除与归档 | `data:maintenance` |
| 操作审计 | `data:audit:view` |

### 7.3 子菜单功能边界

| 子菜单 | 放入内容 | 不放入内容 |
|--------|----------|------------|
| 项目总览 | 项目查询、新建、编辑、状态、数据概览、快捷入口 | 大文件上传、导入失败明细、审计日志全文 |
| 项目导入 | 单项目导入操作、路网前置提示、导入结果摘要 | 跨项目记录查询、归档/删除审批 |
| 导入记录 | 流水查询、失败明细、错误报告下载 | 项目主数据编辑、模板维护 |
| 数据质量 | 质量指标、异常列表、质量报告导出 | 原始上传入口、危险删除操作 |
| 模板与规范 | 模板下载、字段说明、示例文件、导入顺序说明 | 真实业务数据操作 |
| 清除与归档 | 清除预览、清除执行、归档、恢复、删除 | 普通导入操作、普通用户查看入口 |
| 操作审计 | 操作日志、操作人、前后快照、原因 | 业务数据编辑 |

### 7.4 推荐菜单层级

```text
数据管理
├── 项目总览
├── 项目导入
├── 导入记录
├── 数据质量
├── 模板与规范
├── 清除与归档
└── 操作审计
```

若一期不希望一次性实现全部子菜单，可按优先级分阶段上线：

| 阶段 | 子菜单 | 说明 |
|------|--------|------|
| 第一阶段 | 项目总览、项目导入、导入记录、模板与规范 | 先解决用户找项目、导入、查记录、看格式的问题。 |
| 第二阶段 | 数据质量、清除与归档 | 补齐质量核查和危险操作管理。 |
| 第三阶段 | 操作审计 | 完成审计、安全、追责能力。 |

### 7.5 项目列表页

建议结构：

1. 顶部筛选区：项目名称、项目状态、年度、负责人、数据来源单位。
2. 概览统计条：项目总数、可用项目数、最近失败导入数、待核查项目数。
3. 项目表格：
   - 项目名称
   - 状态
   - 数据概览：路线/路段/病害
   - 最近导入：类型、状态、时间
   - 负责人/来源单位
   - 创建/更新时间
   - 操作：导入、查看 GIS、详情、更多
4. 更多菜单：
   - 编辑
   - 清除数据
   - 归档
   - 删除

### 7.6 项目导入页

建议从单纯 Tab 上传升级为“项目数据工作台”：

1. 项目摘要区：项目名称、状态、负责人、数据覆盖概览、最近导入结果。
2. 数据概览区：路网、路段、病害三类卡片，各自展示数量、最后导入时间、质量状态。
3. 导入操作区：路网、路段、病害分区上传，带模板/说明/清除。
4. 导入记录区：默认按时间倒序，支持类型、状态筛选。
5. 记录详情抽屉：展示摘要、警告、失败明细、原始 JSON、操作日志。

项目导入页虽然可从“项目导入”子菜单直接进入，但仍应要求先选择项目。若用户从“项目总览”点击某项目进入，则自动带入项目上下文；若用户直接点击“项目导入”子菜单，则先展示项目选择器，选择后再进入导入工作台。

### 7.7 导入记录页

导入记录从项目导入页中拆出为独立子菜单后，应支持跨项目查询：

| 筛选项 | 说明 |
|--------|------|
| 项目 | 支持按项目名称搜索或下拉选择。 |
| 导入类型 | 路网、路段、病害。 |
| 状态 | 成功、失败、处理中。 |
| 时间范围 | 按开始时间筛选。 |
| 上传人 | 审计和排错使用。 |

列表列建议：项目名称、导入类型、文件名、状态、开始时间、耗时、上传人、结果摘要、操作。

### 7.8 记录详情

导入记录建议增加“详情”按钮，打开抽屉：

- 基础信息：项目、类型、状态、文件、上传人、耗时。
- 结果统计：新增、更新、跳过、失败、警告。
- 质量检查：路线匹配率、空间范围、空值情况。
- 失败明细：行号、字段、原因、修复建议。
- 技术信息：traceId、接口、异常堆栈摘要，仅管理员/运维可见。

## 9. 风险与口径统一

### 9.1 删除语义

当前代码和页面文案偏向：

- 清除：物理删除项目下已归属业务数据；`ALL` 时物理删除导入流水；保留项目主档。
- 删除：先执行 `ALL` 清除，再软删项目主档。

旧规格中仍有部分“业务数据软删”的描述。后续必须二选一：

| 方案 | 优点 | 缺点 |
|------|------|------|
| 维持物理删除 | 数据彻底清理，重新导入简单 | 审计、恢复和误删保护弱 |
| 改为软删业务数据 | 可审计、可恢复 | 数据量增长，需要查询统一过滤 |

建议：生产场景优先采用软删或至少保留删除前快照；演示/本地环境可保留物理删除。

### 9.2 项目依赖关系

路段、病害可能依赖路网进行路线关联。清除单类数据时应提示：

- 清除路网后，保留的路段/病害可能无法在 GIS 中完整关联路线。
- 清除路段后，评定专题和空间统计可能不完整。
- 清除病害后，病害图层和病害分析为空，但路网/路段仍可用。

## 10. 优先级拆分

### P0：先修正“会误导或有风险”的问题

1. 统一清除/删除口径：文档、接口注释、页面文案保持一致。
2. 将“数据管理”拆成父子菜单，至少先落地项目总览、项目导入、导入记录、模板与规范。
3. 清除/删除前增加影响数量预览。
4. 路段/病害导入前强制校验当前项目已有路网。
5. 导入记录结果摘要从 JSON 文本改成人可读统计。
6. 项目列表增加数据概览：路线、路段、病害、最近导入状态。
7. 导入失败展示结构化错误明细，至少支持行号/原因。

### P1：补齐角色协作能力

1. 项目编辑：负责人、年度、来源单位、备注。
2. 权限分级：查看、导入、清除、删除分开控制。
3. 数据质量子菜单：路线匹配率、未分类病害、空几何等。
4. 清除与归档子菜单。
5. 导入记录详情抽屉。

### P2：提升治理和运营能力

1. 项目归档/恢复。
2. 操作审计子菜单和审计日志。
3. 删除审批或输入项目名确认。
4. 导入任务进度、后台异步任务化。
5. 多项目对比和批量导出。

## 11. 建议验收清单

1. 普通用户能在项目列表看出每个项目是否已具备 GIS 可用数据。
2. 管理员执行清除/删除前能看到准确影响范围，并能区分清除与删除。
3. 数据人员导入失败后能定位到文件、行号、字段和原因。
4. 运维人员能通过 traceId 从前端错误定位后端日志。
5. 审计人员能追踪项目创建、导入、清除、删除的操作人和时间。
6. 文档、页面文案、后端注释对“物理删除/软删”的描述一致。
7. GIS 一张图能明确显示当前项目，并在无项目或无数据时给出清晰提示。
8. 当前项目未导入路网时，路段和病害上传入口不可用；直接调用后端导入接口也会返回“请先导入路网数据”。
9. 左侧导航中“数据管理”为可展开/折叠父菜单，子菜单按项目总览、项目导入、导入记录、数据质量、模板与规范、清除与归档、操作审计展示。
10. 权限不足的用户看不到无权限子菜单；进入任一子菜单时父菜单保持展开且当前项高亮。
