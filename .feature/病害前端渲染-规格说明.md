# 病害前端渲染 — 详细功能规格说明

> **依据计划**：`srmp/.plan/病害前端渲染.plan.md`。  
> **文档目的**：将“地图病害数据展示简易方案”扩展为可交付的产品、前端、后端与验收规格，供工程师或大模型直接据此拆分任务与实现。  
> **核心原则**：**地图看到哪里，就查询哪里；地图缩放到合适级别，才展示详细病害。**

---

## 1. 背景与问题

地图页面需要同时展示两类数据：

1. **路网数据**：道路路线、路段、评定单元等基础空间数据。
2. **病害数据**：裂缝、坑槽、沉陷、车辙等挂接在道路空间上的病害点位或病害对象。

路网数据通常作为底层空间上下文展示，数据量相对可控；病害数据可能随年份、路线、检测批次、道路等级和区域快速增长。如果前端一次性加载全部病害并渲染，会带来明显风险：

- 接口响应时间过长，首屏等待不可控。
- 浏览器需要解析大量 JSON，主线程被阻塞。
- 地图引擎同时管理大量 marker / feature，拖动和缩放明显卡顿。
- 内存持续上涨，严重时浏览器标签页崩溃。
- 用户在低缩放层级看到大量重叠病害，信息不可读。

因此，病害数据展示不能采用“一次性查询全部并全部上图”的方式，而应按地图视野、缩放层级和最大返回数量进行分级加载。

---

## 2. 目标与范围

### 2.1 本期目标

本期实现地图病害图层的按需加载与分级渲染：

- 只查询当前地图可视范围内的病害数据。
- 用户拖动或缩放地图后，前端重新计算视野范围并刷新病害图层。
- 通过防抖避免地图连续变化时频繁请求接口。
- 根据地图缩放级别决定展示模式：
  - 低缩放：不展示明细，只展示范围统计或提示。
  - 中缩放：展示聚合点。
  - 高缩放：展示具体病害点或病害对象。
- 后端控制接口最大返回数量，避免单次响应过大。
- 当前区域数据过多时返回明确提示，引导用户放大地图。

### 2.2 非目标

以下内容不作为本期强制范围：

- 离线瓦片化病害图层。
- WebGL 大规模矢量切片渲染。
- 病害编辑、删除、派单和处置闭环。
- 病害详情完整业务页面重构。
- 全量历史病害时空动画播放。

如后续数据规模继续增长，可在本方案基础上演进到服务端矢量切片、瓦片聚合或专用空间检索服务。

---

## 3. 总体方案

### 3.1 核心策略

采用：

```text
地图范围查询 + 缩放级别控制 + 最大返回数量限制 + 前端防抖
```

即：

1. 前端监听地图视野变化。
2. 在拖动或缩放结束后，延迟一小段时间发起查询。
3. 查询参数携带当前地图边界和缩放级别。
4. 后端只查询边界框内的病害数据。
5. 后端根据 `zoom` 和数据密度返回统计、聚合或明细。
6. 前端根据 `mode` 选择对应渲染方式。

### 3.2 基本流程

```text
用户打开地图
    ↓
加载路网底图 / 路网图层
    ↓
获取当前地图视野范围和缩放级别
    ↓
请求当前范围内的病害数据
    ↓
后端按 bbox + zoom 查询和裁剪
    ↓
后端返回 summary / cluster / detail / too_many
    ↓
前端清理旧病害图层并渲染新数据
    ↓
用户拖动或缩放地图后重新触发
```

## 4. 地图交互规格

### 4.1 触发时机

病害数据加载不应在地图移动过程中连续触发，应在以下事件完成后触发：

- 地图初始化完成。
- 用户拖动地图结束。
- 用户缩放地图结束。
- 用户切换病害图层开关为开启。
- 用户修改与病害相关的筛选条件，例如年份、路线、病害类型、严重程度。

建议监听地图引擎的 `moveend`、`zoomend` 或等价事件。如果地图引擎只有统一 `moveend`，可在事件中读取最新 zoom。

### 4.2 请求防抖

每次地图变化后延迟发起请求，建议防抖时间为 **300ms**。

示例逻辑：

```javascript
let diseaseLoadTimer = null

function onMapViewportChanged() {
  clearTimeout(diseaseLoadTimer)

  diseaseLoadTimer = setTimeout(() => {
    loadDiseaseData()
  }, 300)
}
```

要求：

- 新事件到来时取消上一次未执行的定时器。
- 若上一次请求仍未返回，应支持取消请求或丢弃旧响应。
- 用户连续拖动地图时，最终只以最后一次视野为准。

### 4.3 旧请求处理

前端必须避免旧请求覆盖新视野数据。

推荐做法：

1. 每次请求生成递增 `requestSeq`。
2. 响应返回后判断该响应是否仍是最新请求。
3. 如果不是最新请求，直接丢弃，不更新图层。

示例：

```javascript
let latestDiseaseRequestSeq = 0

async function loadDiseaseData() {
  const seq = ++latestDiseaseRequestSeq
  const params = buildDiseaseQueryParams()
  const result = await fetchDiseases(params)

  if (seq !== latestDiseaseRequestSeq) {
    return
  }

  renderDiseaseLayer(result)
}
```

如果请求库支持 `AbortController` 或取消 token，可在新请求发出前取消上一请求。

### 4.4 图层开关

病害图层应具备独立开关：

- 关闭时：
  - 清空当前病害图层。
  - 不再响应地图移动产生的病害查询。
  - 正在进行的请求返回后不得重新渲染。
- 开启时：
  - 立即按当前地图视野请求一次病害数据。
  - 后续继续跟随地图移动刷新。

## 5. 查询参数规格

### 5.1 必填参数

前端每次请求至少携带：

| 参数 | 类型 | 说明 |
|------|------|------|
| `minLng` | number | 当前地图视野最小经度 |
| `minLat` | number | 当前地图视野最小纬度 |
| `maxLng` | number | 当前地图视野最大经度 |
| `maxLat` | number | 当前地图视野最大纬度 |
| `zoom` | number | 当前地图缩放级别 |

示例：

```http
GET /api/map/diseases?minLng=113.1&minLat=22.1&maxLng=114.1&maxLat=23.1&zoom=14
```

### 5.2 可选筛选参数

建议预留以下参数，便于与现有一张图筛选能力结合：

| 参数 | 类型 | 说明 |
|------|------|------|
| `year` | number | 检测年份或评定年份 |
| `routeCode` | string | 路线编码 |
| `diseaseType` | string | 病害类型 |
| `diseaseName` | string | 病害名称 |
| `severity` | string | 严重程度 |
| `projectId` | string | 数据管理项目 ID |
| `tenantId` | string | 租户 ID，通常也可由请求头传递 |

### 5.3 参数校验

后端必须校验：

- `minLng < maxLng`
- `minLat < maxLat`
- 经度范围在 `[-180, 180]`
- 纬度范围在 `[-90, 90]`
- `zoom` 为有效数字
- bbox 面积不能异常大到覆盖全球，必要时返回 `too_many` 或参数错误

若参数非法，返回明确错误，前端展示轻量提示，不应导致地图页面崩溃。

## 6. 缩放级别展示策略

### 6.1 默认策略

| zoom 范围 | 后端返回模式 | 前端展示方式 |
|-----------|--------------|--------------|
| `zoom < 12` | `summary` | 不展示明细，仅展示当前范围病害总数或提示 |
| `12 <= zoom < 15` | `cluster` | 展示聚合点 |
| `zoom >= 15` | `detail` | 展示具体病害 |

### 6.2 低缩放：统计模式

低缩放级别下不渲染具体病害点，避免地图上出现大量重叠图标。

前端展示建议：

- 在地图角落或图层状态栏展示：“当前视野病害 N 条，放大地图查看详情”。
- 不在地图上绘制大量 marker。
- 可选：按行政区、网格或路线显示粗粒度数量标签。

返回示例：

```json
{
  "mode": "summary",
  "total": 23560,
  "message": "当前缩放级别较低，请放大地图查看病害详情",
  "data": []
}
```

### 6.3 中缩放：聚合模式

中缩放级别下展示聚合点，聚合点表示某个区域内的病害数量。

返回示例：

```json
{
  "mode": "cluster",
  "total": 2380,
  "data": [
    {
      "lng": 113.25,
      "lat": 23.18,
      "count": 56
    }
  ]
}
```

前端渲染要求：

- 聚合点显示数量。
- 聚合点颜色或大小可随 `count` 增大而增强。
- 点击聚合点时：
  - 默认行为：放大一级或两级，并以该聚合点为中心。
  - 若业务需要，也可弹出该聚合区域统计。

### 6.4 高缩放：明细模式

高缩放级别下展示具体病害点。

返回示例：

```json
{
  "mode": "detail",
  "total": 128,
  "data": [
    {
      "id": "D001",
      "type": "裂缝",
      "name": "横向裂缝",
      "severity": "严重",
      "lng": 113.25123,
      "lat": 23.18123,
      "routeCode": "G210",
      "stake": "K12+300",
      "year": 2026
    }
  ]
}
```

前端渲染要求：

- 每个病害对象渲染为地图点或业务图标。
- 图标样式应区分严重程度，至少支持正常 / 中度 / 严重等视觉差异。
- 鼠标悬停或点击展示简要信息。
- 点击病害点后，可联动右侧详情面板或弹窗，展示病害名称、路线、桩号、严重程度等。

## 7. 返回数量控制

### 7.1 最大返回条数

后端必须限制单次接口返回数量。默认建议：

```text
detail 最大返回 1000 条
cluster 最大返回 500 个聚合点
```

如果当前视野内病害数量超过明细返回上限，即使 `zoom >= 15`，也不应强行返回全部数据。

### 7.2 数据过多模式

当当前区域数据过多时，返回：

```json
{
  "mode": "too_many",
  "total": 15230,
  "message": "当前区域数据太多，请放大地图查看",
  "data": []
}
```

前端处理：

- 清空旧明细图层或保留最近一次有效数据并标记过期，具体选一种并保持一致。
- 在地图上方或角落展示提示。
- 不渲染过多 marker。
- 引导用户放大地图或增加筛选条件。

## 8. 后端查询规格

### 8.1 查询原则

后端不得查询全部病害后再在内存中裁剪。必须在数据库层使用空间条件过滤当前 bbox。

错误方式：

```text
select * from disease_record
```

正确方式：

```text
select * from disease_record
where geom within current bbox
limit N
```

### 8.2 数据库存储建议

如果病害数据存储在 MongoDB 中，建议使用 GeoJSON `Point`：

```json
{
  "id": "D001",
  "type": "裂缝",
  "severity": "严重",
  "location": {
    "type": "Point",
    "coordinates": [113.25123, 23.18123]
  }
}
```

创建空间索引：

```javascript
db.disease.createIndex({ location: "2dsphere" })
```

如果病害数据存储在 PostgreSQL / PostGIS 中，建议使用 `geometry(Point, 4326)` 或等价空间列，并建立 GiST 索引：

```sql
create index if not exists idx_disease_record_geom
on disease_record
using gist (geom);
```

查询时使用空间相交或包含条件，例如：

```sql
where ST_Intersects(
  geom,
  ST_MakeEnvelope(:minLng, :minLat, :maxLng, :maxLat, 4326)
)
```

### 8.3 聚合算法建议

聚合模式可以采用网格聚合：

1. 根据 zoom 计算网格大小。
2. 将 bbox 内病害按网格分组。
3. 每个网格返回中心点和数量。

聚合点字段：

| 字段 | 说明 |
|------|------|
| `lng` | 聚合点中心经度 |
| `lat` | 聚合点中心纬度 |
| `count` | 聚合数量 |
| `severityStats` | 可选，严重程度统计 |
| `typeStats` | 可选，病害类型统计 |

### 8.4 排序规则

明细模式建议按以下优先级排序：

1. 严重程度高的优先。
2. 当前业务年份优先。
3. 距离地图中心更近的优先。
4. 创建时间或检测时间较新的优先。

如果暂不实现复杂排序，至少保证返回结果稳定，例如按 `id` 或 `created_at desc`。

## 9. 前端渲染规格

### 9.1 图层结构

建议前端维护独立病害图层：

```text
diseaseLayer
  ├── summary state
  ├── cluster layer
  └── detail layer
```

每次新数据返回后：

1. 判断 `mode`。
2. 清理与当前模式不匹配的旧图层。
3. 渲染新数据。
4. 更新图层状态栏。

### 9.2 状态栏信息

地图页面应显示病害图层当前状态：

| 状态 | 展示示例 |
|------|----------|
| 加载中 | `病害加载中...` |
| 统计模式 | `当前范围病害 23560 条，请放大查看详情` |
| 聚合模式 | `当前范围病害 2380 条，已聚合显示` |
| 明细模式 | `当前范围病害 128 条` |
| 数据过多 | `当前区域数据太多，请放大地图查看` |
| 请求失败 | `病害数据加载失败，请稍后重试` |

### 9.3 图标规则

具体病害图标建议按严重程度区分：

| 严重程度 | 建议样式 |
|----------|----------|
| 轻微 / 低 | 黄色或浅橙色 |
| 中等 / 中 | 橙色 |
| 严重 / 高 | 红色 |
| 未知 | 灰色 |

不同病害类型可通过图标形状或二级标识区分，但本期不强制。

### 9.4 点击交互

点击具体病害点：

- 高亮当前点。
- 展示病害摘要：
  - 病害名称
  - 严重程度
  - 路线编码
  - 桩号
  - 年份
  - 坐标
- 可选：触发右侧详情面板加载完整详情。

点击聚合点：

- 默认放大并移动到聚合点中心。
- 放大后触发下一轮病害查询。

## 11. 加载、错误与空状态

### 11.1 加载中

病害查询期间：

- 不应阻塞地图基础操作。
- 可在图层开关旁展示 loading。
- 若用户继续移动地图，旧请求应被取消或响应后丢弃。

### 11.2 空数据

当前视野没有病害时：

```json
{
  "mode": "detail",
  "total": 0,
  "data": [],
  "message": "当前范围暂无病害数据"
}
```

前端展示：

- 清空病害图层。
- 显示轻量提示，不使用错误样式。

### 11.3 请求失败

接口失败时：

- 保留或清空旧图层需统一策略。建议保留旧图层并显示“数据可能不是最新”的弱提示。
- 控制台记录完整错误。
- 页面提示使用简短文案，不暴露长堆栈。

## 12. 性能要求

### 12.1 前端性能

| 指标 | 要求 |
|------|------|
| 防抖时间 | 默认 300ms |
| 明细 marker 数 | 单次不超过 1000 |
| 聚合点数 | 单次不超过 500 |
| 连续拖动 | 不产生请求风暴 |
| 旧响应 | 不覆盖新视野数据 |

### 12.2 后端性能

| 指标 | 要求 |
|------|------|
| bbox 查询 | 必须命中空间索引 |
| 明细查询 | 必须有 limit |
| 聚合查询 | 不返回原始全量数据 |
| 数据过多 | 返回 `too_many` 或聚合，不直接返回明细 |

### 12.3 网络负载

响应数据应尽量只返回渲染需要字段。明细列表不应默认返回大字段，例如完整图片列表、大文本备注、历史处置记录等。完整详情可在用户点击病害后另行查询。

---

## 13. 安全与多租户

如果系统存在租户或项目隔离，后端必须在 bbox 条件之外叠加权限条件：

- 当前租户 ID。
- 当前项目 ID。
- 用户可访问的路线或行政区域。
- 数据软删除标记，例如 `deleted = false`。

前端传入的 bbox 只表示空间范围，不代表访问权限。

## 14. 验收标准

### 14.1 功能验收

- 打开地图后，病害图层按当前视野请求数据。
- 拖动地图结束后，病害数据刷新为新视野数据。
- 缩放地图后，返回模式随 zoom 变化：
  - `zoom < 12` 显示统计或提示。
  - `12 <= zoom < 15` 显示聚合点。
  - `zoom >= 15` 显示具体病害。
- 当前区域数据过多时，不渲染海量 marker，并提示用户放大地图。
- 当前区域无数据时，清空病害图层并显示空状态。
- 点击聚合点可以放大地图。
- 点击病害点可以看到病害摘要。

### 14.2 性能验收

- 连续拖动地图 5 秒内，不出现明显页面卡顿。
- 连续拖动过程中不会产生大量并发病害请求。
- 单次明细返回超过限制时，后端不会返回全部数据。
- 浏览器内存不会因多次移动地图持续上涨。

### 14.3 接口验收

- bbox 参数非法时返回明确错误。
- bbox 查询仅返回当前视野范围内数据。
- 数据库空间索引存在并被查询使用。
- 返回结构包含 `mode`、`total`、`data`。
- `too_many` 模式下 `data` 为空或仅包含轻量提示数据。

## 15. 推荐实施步骤

1. 后端新增或改造病害地图查询接口，支持 bbox、zoom 和 limit。
2. 后端为病害空间字段增加空间索引。

4. 前端封装病害查询 API。
5. 前端在地图初始化、拖动结束、缩放结束后触发防抖加载。
6. 前端实现旧请求丢弃或取消。
7. 前端实现聚合点和明细点渲染。
8. 前端实现加载、空数据、数据过多、请求失败状态。
9. 使用高数据量场景进行性能验证。

## 16. 总结

本方案避免一次性加载全部病害数据，通过当前地图视野和缩放级别控制查询与渲染规模。后端负责空间过滤、聚合和数量限制，前端负责防抖、请求时序控制和分模式渲染。

最终目标是：

```text
在病害数据量较大时，地图仍能稳定、流畅、可读地展示用户当前真正需要看的内容。
```
