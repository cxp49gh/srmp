# 导入路段数据 — AI 实现规格书（Shapefile/TAR → 四级分表）

> **读者模型**：本文件供大模型或工程师直接据此生成/修改代码，不要求再向人类澄清业务意图。  
> **硬性规则**：凡标注 **【已定】** 的条目按本文实现，勿自行改语义；标注 **【可选】** 的可按注释取舍。  
> **关联计划**：`srmp/.plan/导入路段数据.plan.md`  
> **本期范围（相对旧版变更）**：**【已定】** 路段包导入须解析并入库 **四类** Shapefile：**线路级、台账级、公里级、百米级**；数据库 **四张物理表一一对应** 存储，**表名与级别语义对齐**（见 §1.3）。不再采用「仅处理两级、其余忽略」策略。  
> **兼容与迁移**：若仓库仍存在历史表 `road_section`、`road_evaluation_unit`，实现阶段须给出 **迁移/双写/视图兼容** 策略并在 DDL 中落地 **【已定】**。

## 0. 执行摘要（给模型的 TL;DR）

1. **后端主归属**：**【已定】** 解压、GeoTools 读 shp、租户上下文、四级分表写库，放在 **`srmp-road-asset`**（包根 `com.smartroad.srmp.roadasset`）。  
2. **四张表**：**【已定】** 仅使用下列四表承载四类导入几何与属性（名称与级别一一对应，勿混用一张表多级别）：  
   - **`road_section_line`** — 线路级  
   - **`road_section_ledger`** — 台账级  
   - **`road_section_km`** — 公里级  
   - **`road_section_hm`** — 百米级  
3. **评定结果**：`assessment_result` 仍归属 **`srmp-assessment`**；**【已定】** `object_type` / `object_id` 须能区分上述四级（见 §7），或通过 **扩展 `object_type` 枚举** + 四表主键引用。  
4. **几何**：与路网导入一致，入库 **EPSG:4326** 的 **2D** `LineString`（各级均为线要素时）；若某级为点/面，在实现 PR 中单开附录 **【可选】**。  
5. **事务**：**【已定】** 单次上传 **整包原子**：任一级任一要素校验失败或任一行写库失败 → **全部回滚**（含 `assessment_result`）。  
6. **前端**：数据管理编排导入或独立 REST 入口；上传 **`.tar`**，成功后刷新 GIS 中 **与四级对应的图层或合并数据源**（实现 PR 锁定一种加载策略）。  
7. **本文档**：规格说明；DDL 轮廓与字段为 **【已定】** 方向性要求，具体长度/索引以实现 PR 中 `srmp_full_init.sql` 为准。

## 1. 仓库事实与表模型 **【已定】**

### 1.0 模块边界

| 范围 | 要求 |
|------|------|
| 解压、Shapefile、字符集、坐标变换 | **`srmp-road-asset`**，复用路网导入 GeoTools、`commons-compress`。 |
| 四级表 CRUD / Mapper | **`srmp-road-asset`**：新建 `RoadSectionLine` / `RoadSectionLedger` / `RoadSectionKm` / `RoadSectionHm`（类名可微调，须与四表一一对应）。 |
| `assessment_result` | **`srmp-assessment`**：经 `AssessmentResultService` 写入；禁止在 road-asset 手写跨模块 SQL。 |
| `srmp-admin` | 启动、`multipart`、**【可选】** `srmp.import.section.dbf-charset`。 |

### 1.1 与 `road_route` 的关系 **【已定】**

| 依赖 | 要求 |
|------|------|
| `road_route` | 四类导入均按 **当前租户** `tenant_id` + **路线编号** 解析 `route_id`（见 §4.2 匹配规则）。**【已定】** 若在路网中**匹配不到**对应 `route_code`：**仍写入**该级要素，`route_id` 置空，`route_code` 写入 SHP 解析得到的路线编号（或 §6 自动编码所用的同一套「入库路线编号」）；导入结果 **`warnings`** 中汇总提示「路线编号未在路网登记」类文案，便于补录路网后再次关联。**命中路网**时：`route_id` 填 `road_route.id`，**`route_code` 一律写库内规范值**（`road_route.route_code`，与路网导入一致），含自动生成的 `line_code` / `ledger_code` / `km_code` / `hm_code` 前缀及 `assessment_result.route_code`。 |

### 1.2 四表命名与语义（与级别一一对应）**【已定】**

| 产品级别 | 物理表名 | 用途简述 |
|----------|----------|----------|
| 线路级 | **`road_section_line`** | 沿路线布设的 **线路级** 路段几何与属性（原「路线级 / road_section」语义迁入本表）。 |
| 台账级 | **`road_section_ledger`** | **台账级** 单元（原「台账级 / road_evaluation_unit」粒度迁入本表；字段与桩号区间、宽度、路面等台账字段对齐）。 |
| 公里级 | **`road_section_km`** | **公里级** 离散或连续单元（如按整公里桩区间一条记录；具体键与 `section_code`/`unit_code` 生成见 §6）。 |
| 百米级 | **`road_section_hm`** | **百米级** 单元（百米桩区间；键生成规则见 §6）。 |

**命名约定**：统一前缀 **`road_section_`** + 级别后缀 **`line` / `ledger` / `km` / `hm`**，便于检索与权限脚本；**禁止**继续用单表 `road_section` 混存多级（除非过渡期 **【可选】** 使用 **视图** `road_section` → 指向 `road_section_line`）。

### 1.3 四表公共字段轮廓 **【已定】**

四表均建议包含（与现网 `BaseEntity` 风格一致）：`id`, `tenant_id`, `route_id`, `route_code`, `direction`, `start_stake`, `end_stake`, `geom`, `project_id`（数据管理编排导入时）, `created_at`, `updated_at`, `deleted`, `created_by`, `updated_by`。

**各级差异字段（方向性，实现 PR 可微调）**：

| 列族 | `road_section_line` | `road_section_ledger` | `road_section_km` | `road_section_hm` |
|------|----------------------|------------------------|---------------------|---------------------|
| 业务编码唯一键 | `line_code`（租户内唯一，替代原 `section_code` 语义） | `ledger_code` | `km_code` | `hm_code` |
| 名称/描述 | `line_name` 等 | `ledger_name` 等 | **【可选】** `km_label` | **【可选】** `hm_label` |
| 长度 | `length_km` | `length_m` | `length_m` 或 `length_km` **【已定】** 与 shp 语义一致即可 | 同左 |
| 路面/技术 | `pavement_type`, `technical_grade`, `road_width`, `lane_count`… | 同族字段与台账一致 | **【可选】** 按数据源裁剪 | **【可选】** |
| 关联 | 可空 `parent_line_id` **【可选】** | 可空 `line_id` → `road_section_line.id` **【已定】** 推荐外键语义，便于从线→台账追溯 | 可空 `ledger_id` / `line_id` **【已定】** 实现 PR 二选一文档化 | 可空指向 `km` 或 `line` **【已定】** 实现 PR 文档化 |

**唯一约束建议**：各表 `UNIQUE(tenant_id, xxx_code)`；若与 `project_id` 组合唯一，采用 **`UNIQUE(tenant_id, project_id, xxx_code)`**（与数据管理项目域一致）**【可选】** 由产品定。

## 3. 文件格式与级别识别

### 3.1 TAR 包结构 **【已定】**

- 非压缩 `.tar`；递归查找 `.shp`。  
- 每组 `{base}.shp` + `dbf` + `shx` + **`prj` 必填**。  
- **【已定】** 至少存在 **四组** 可识别为四级的 shp（或 PB-5 **【可选】** 允许某级 0 要素）。

### 3.2 DBF 字符集 **【已定】**

- `srmp.import.section.dbf-charset`（默认 UTF-8）；列名 **trim**；支持中英别名（§4.3 可沿用并扩展）。

### 3.3 级别识别（基名或路径，不区分大小写）**【已定】**

| 级别 | 归入本级的匹配（满足任一即可；实现须收敛为稳定词表并写单测） |
|------|--------------------------------------------------------------|
| **线路级** | `线路级`、`线路`、`road_section_line`、`section_line`、`xlj`（示例，实现时固定英文关键字优先）。 |
| **台账级** | `台账级`、`台账`、`road_section_ledger`、`ledger`、`评定单元`、`evaluation`。 |
| **公里级** | `公里级`、`公里`、`road_section_km`、`km_unit`、`glj`。 |
| **百米级** | `百米级`、`百米`、`road_section_hm`、`hm_unit`、`bmj`。 |

**无法识别**：**【已定】** 若存在 shp 既不匹配四级也不匹配忽略白名单 → 整包失败：`存在无法识别的 Shapefile 级别，请按命名约定放置四级数据`。

**处理顺序**：见 PB-8。

## 4. 属性列与数据库映射

### 4.1 逻辑列 → 四表与 `assessment_result`

**【已定】** 四类 shp **逻辑列集合可对齐**（与旧版 §4.1 思路一致），但 **物理落表按级别分流**：

| 逻辑列含义 | `road_section_line` | `road_section_ledger` | `road_section_km` | `road_section_hm` | `assessment_result` |
|------------|---------------------|------------------------|---------------------|---------------------|---------------------|
| 路线编号 | ✓ | ✓ | ✓ | ✓ | ✓ |
| 起止桩号 | ✓ | ✓ | ✓ | ✓ | ✓ |
| 检测方向 | ✓ | ✓ | ✓ | ✓ | ✓ |
| 技术等级/路面/宽度/长度 | ✓ | ✓ | **【可选】** 裁剪 | **【可选】** 裁剪 | — |
| MQI/PQI/… | — | — | **【可选】** | **【可选】** | ✓（按写入约定） |
| 检测年度 | — | — | — | — | ✓ 必填 |

**DR/IRI 等扩展列**：仍可按旧版约定进入 `zero_reason` 或后续加列 **【可选】**。

### 4.2 路线编号：路网 `route_code` ↔ 路段 `linkCode` **【已定】**

**术语对照**（勿混淆**列名**与**库字段**）：

| 侧 | 载体 | 列名 / 字段名 | 含义 |
|----|------|----------------|------|
| **路网** | 表 `road_route` | **`route_code`** | 路线编号（主数据） |
| **路网** | 路网 Shapefile | **`ROUTE_NO`** | 导入时写入上表 **`route_code`**（见《导入路网数据》） |
| **路段** | 路段包 Shapefile | **`linkCode`** | 与上表 **`route_code` 同义的路线编号**（检测包标准列名） |
| **路段** | 四张路段表 / 评定 | **`route_code`** | **命中路网**时为库内规范值；未命中时为 `linkCode` 解析值 |

**`linkCode` 的取值须能与 `road_route.route_code` 按下方 §4.2.2 规则匹配上**（通常二者字符串一致或可剥后缀后一致），`route_id` 才能非空。

#### 4.2.1 DBF 读取顺序 **【已定】**

**标准**：**`linkCode`**（大小写不敏感，如 `LINKCODE`）。

**兼容**（仅当 `linkCode` 缺或空时按序择一）：`ROUTE_NO`、`routeCode`、`ROUTE_CODE`、`路线编码`、`route_code` 等——便于与路网 SHP 同结构数据混用，**非检测包模板必备列**。

#### 4.2.2 与路网匹配 **【已定】**

- 实现：`RouteCodeNetworkLookup`（`resolveBatch` / `lookup`）。
- 对 **由 `linkCode`（或兼容列）解析出的字符串** 依次尝试 **原文**、**剥末尾上下行后缀**（`AB`、`A`、`B` 等），在 **`tenant_id` + `road_route.route_code` 精确等值** 且 `deleted = false` 下查询。
- **命中**：`route_id` = 命中行 `id`；**业务表与 `assessment_result` 的 `route_code`** 及 **§6 的 `{route_code}`** = **`road_route.route_code`**。
- **未命中**：`route_id` = `NULL`；入库 `route_code` = 解析值；`warnings` 提示。

**常见关联失败**：`linkCode` 填了**链路 ID** 而路网存的是**行政路线编号**；租户不一致；与 `route_code` **大小写/空格**不一致（PostgreSQL `=` 区分大小写）。

### 4.3 英文别名

沿用旧版 §4.2 思路，并增加：`KM`、`HM`、`KM_UNIT`、`HM_UNIT` 等用于级别无关字段的别名表 **【可选】** 单列附录。

## 12. 测试用例建议

1. 最小包：四级各 1 条要素 → 四表各 1 行 + `assessment_result` 行数符合指标列。  
2. 缺「百米级」组且未启用 PB-5 可选 → 失败。  
3. **`route_code` 在路网无匹配**：**【已定】** 仍成功入库，`route_id` 为空，响应含 `warnings`；路网预置命中时 **`route_code` 与 `road_route.route_code` 一致**（见 §4.2）。  
4. 重复 `hm_code` → 失败。  
5. 字符集 UTF-8 / GBK。  
6. **迁移**：旧 `road_section` 数据可导入或迁移至 `road_section_line` 后校验条数一致 **【可选】**。
