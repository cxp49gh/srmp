# 数据管理页面父子菜单改造 — 实施计划

> 生成日期：2026-05-19  
> 依据文档：`.feature/数据管理页面-角色视角整理与优化建议.md`  
> 执行目标：把当前单一“数据管理”入口拆成可折叠父菜单，并按功能拆分为多个子菜单页面。本文面向后续执行模型，要求严格按步骤推进，不允许跳步大改。

---

## 0. 执行原则

1. 每一步只做本步骤声明的改动，不顺手重构无关页面。
2. 先做菜单和路由，再做页面拆分，再做接口增强，最后做权限、审计、质量等治理能力。
3. 所有新页面必须复用现有后台壳组件和交互风格，优先复用 `AgentLayout`、`AgentPageShell`、现有 Element Plus 表格/表单模式。
4. 路段、病害导入必须以“项目已导入路网”为前置条件。前端要提示，后端最终也要校验。
5. 危险操作必须放到“清除与归档”子菜单或二级确认流程里，不要继续散落在普通列表主操作区。
6. 每一阶段完成后必须运行前端构建；涉及后端时必须运行对应 Maven 编译或测试。

---

## 1. 目标菜单结构

最终左侧导航结构必须是：

```text
数据管理
├── 项目总览
├── 项目导入
├── 导入记录
├── 数据质量
├── 模板与规范
├── 清除与归档
└── 操作审计
```

父菜单“数据管理”只负责展开/折叠和聚合子菜单。用户点击父菜单时，优先展开/折叠；如果当前侧栏实现必须跳转，则跳转到“项目总览”。

---

## 2. 分阶段实施顺序

### 第一阶段：菜单与基础页面拆分

目标：先让信息架构成立，避免继续把所有能力堆在一个页面。

必须完成：

1. 将“数据管理”改为父菜单。
2. 增加以下子菜单：
   - 项目总览
   - 项目导入
   - 导入记录
   - 模板与规范
3. 保留现有项目列表和项目导入能力，但调整入口位置。
4. 暂不实现“数据质量”“清除与归档”“操作审计”的完整业务，可先放占位页面，页面必须明确标注待实现能力。

验收：

1. 左侧菜单中“数据管理”可以展开/折叠。
2. 进入任一子菜单时，“数据管理”保持展开，当前子菜单高亮。
3. 现有项目列表功能可从“项目总览”进入。
4. 现有项目导入功能可从“项目导入”进入，直接进入时要求用户先选择项目。
5. 前端构建通过。

### 第二阶段：项目总览增强

目标：让普通用户和管理员不用进入导入页，也能判断项目是否可用。

必须完成：

1. 项目总览表格展示：
   - 项目名称
   - 项目状态
   - 路线数
   - 路段数
   - 病害数
   - 最近导入类型
   - 最近导入状态
   - 最近导入时间
   - 创建时间
   - 更新时间
2. 增加项目状态：
   - 未导入
   - 部分导入
   - 可用
   - 导入失败
   - 已清空
3. 操作列保留常用入口：
   - 查看 GIS
   - 导入
   - 详情
4. 危险操作不要直接作为主按钮暴露。清除、删除、归档应移动到“更多”或后续“清除与归档”子菜单。

后端建议：

1. 增加项目数据概览接口：`GET /api/data-mgmt/projects/{id}/summary`。
2. 如果列表需要批量展示统计，增加分页接口直接返回 summary 字段，避免前端 N+1 请求。

验收：

1. 用户在项目总览可以判断每个项目是否具备 GIS 可用数据。
2. 无数据项目点击“查看 GIS”时有明确提示。
3. 有失败导入记录的项目能看到失败状态或最近失败标记。

### 第三阶段：项目导入工作台改造

目标：把导入页从简单 Tab 上传升级为单项目数据工作台。

必须完成：

1. 顶部展示项目摘要：
   - 项目名称
   - 项目状态
   - 路线数
   - 路段数
   - 病害数
   - 最近一次成功导入时间
2. 分区展示三个导入入口：
   - 导入路网
   - 导入路段
   - 导入病害
3. 路网前置规则：
   - 新项目未导入路网时，路段和病害上传按钮禁用。
   - 路网导入失败时，仍禁用路段和病害上传。
   - 路网导入成功且路线数大于 0 时，开放路段和病害上传。
   - 清除路网后，重新禁用路段和病害上传。
4. 每个导入入口都必须提供：
   - 文件类型说明
   - 模板或规范入口
   - 上传按钮
   - 最近一次导入结果摘要
5. 导入成功后刷新项目摘要和导入记录。

后端建议：

1. 在数据管理编排导入接口中校验路网前置条件。
2. 校验口径：当前项目下已归属、未删除的路网路线数大于 0。
3. 失败提示统一为：`请先导入路网数据，再导入路段或病害数据`。

验收：

1. 未导入路网的项目不能导入路段或病害。
2. 直接调用后端路段/病害导入接口也不能绕过路网前置校验。
3. 路网导入成功后，不刷新页面也能开放路段/病害入口。

### 第四阶段：导入记录独立子菜单

目标：让数据人员、运维、审计能跨项目查导入流水和失败原因。

必须完成：

1. 新建“导入记录”页面。
2. 支持筛选：
   - 项目
   - 导入类型
   - 状态
   - 时间范围
   - 上传人
3. 列表展示：
   - 项目名称
   - 导入类型
   - 文件名
   - 状态
   - 开始时间
   - 耗时
   - 上传人
   - 结果摘要
   - 操作
4. 增加记录详情抽屉：
   - 基础信息
   - 结果统计
   - 失败明细
   - 警告信息
   - 技术信息

后端建议：

1. 增加跨项目导入记录分页接口：`GET /api/data-mgmt/import-records/page` 或 `POST /api/data-mgmt/import-records/page`。
2. 增加导入记录详情接口：`GET /api/data-mgmt/projects/{id}/import-records/{recordId}`。

验收：

1. 可查全部项目导入记录。
2. 可按失败状态筛选。
3. 失败记录能定位到文件、行号、字段、原因。

### 第五阶段：模板与规范子菜单

目标：让数据人员导入前就知道文件该怎么准备。

必须完成：

1. 新建“模板与规范”页面。
2. 按类型分组：
   - 路网
   - 路段
   - 病害
3. 每组展示：
   - 支持格式
   - 文件结构要求
   - 必填字段
   - 坐标系要求
   - 示例文件或模板下载
   - 常见错误
4. 必须明确导入顺序：
   - 先导入路网
   - 再导入路段
   - 再导入病害或与路段并行导入病害，但病害仍需依赖路网

验收：

1. 数据人员可以从此页面下载或查看路网、路段、病害模板。
2. 页面明确说明路段和病害不能早于路网导入。

### 第六阶段：数据质量子菜单

目标：补齐“导入成功不等于数据可用”的质量核查能力。

必须完成：

1. 新建“数据质量”页面。
2. 支持按项目查看质量报告。
3. 展示指标：
   - 路线数
   - 路段数
   - 病害数
   - 路线匹配率
   - 未匹配路线数量
   - 未分类病害数量
   - 空几何数量
   - 异常坐标数量
4. 支持导出质量报告。

后端建议：

1. 增加接口：`GET /api/data-mgmt/projects/{id}/quality-report`。

验收：

1. 每个项目都能查看质量报告。
2. 质量异常能定位到具体类型和建议处理方式。

### 第七阶段：清除与归档子菜单

目标：集中管理危险操作，避免误操作。

必须完成：

1. 新建“清除与归档”页面。
2. 支持选择项目。
3. 清除前必须展示影响范围：
   - 路线数
   - 路段数
   - 病害数
   - 导入流水数
4. 支持清除范围：
   - 全部
   - 路网
   - 路段
   - 病害
5. 支持归档项目。
6. 删除项目必须要求二次确认，建议输入项目名称确认。

后端建议：

1. 增加清除预览接口：`POST /api/data-mgmt/projects/{id}/clear-preview`。
2. 增加归档接口：`POST /api/data-mgmt/projects/{id}/archive`。
3. 增加恢复接口：`POST /api/data-mgmt/projects/{id}/restore`。

验收：

1. 所有清除和删除操作都能看到影响范围。
2. 清除路网后，项目导入页重新禁止路段和病害导入。
3. 归档项目默认不出现在普通项目列表，但仍可审计。

### 第八阶段：操作审计子菜单

目标：补齐追责和安全审计能力。

必须完成：

1. 新建“操作审计”页面。
2. 支持筛选：
   - 项目
   - 操作类型
   - 操作人
   - 时间范围
3. 展示字段：
   - 项目名称
   - 操作类型
   - 操作人
   - 操作时间
   - 操作结果
   - 操作原因
   - 前后快照
4. 支持查看详情。

后端建议：

1. 增加审计日志表或复用系统审计日志。
2. 增加接口：`GET /api/data-mgmt/audit-logs/page`。

验收：

1. 能查到项目创建、编辑、导入、清除、删除、归档操作。
2. 危险操作能看到操作前后数据量和操作人。

|------|------|

1. 旧路由 `/admin/data-management` 应重定向到 `/admin/data-management/projects`。
2. 旧路由 `/admin/data-management/:projectId/import` 可以继续保留，避免已有链接失效。

## 5. 执行检查清单

每次执行任务前：

1. 读取 `.feature/数据管理页面-角色视角整理与优化建议.md`。
2. 确认当前要做的是第几阶段。
3. 只修改该阶段允许的文件。
4. 不删除现有可用功能。
5. 保持旧路由兼容。

每次执行任务后：

1. 运行前端构建：`npm run build`。
2. 若修改后端，运行相关 Maven 编译或测试。
3. 检查左侧菜单是否能展开/折叠。
4. 检查当前子菜单是否高亮。
5. 检查无权限菜单是否隐藏或至少预留权限字段。
6. 检查路网前置规则是否仍成立。

---

## 6. 禁止事项

1. 不允许把所有功能继续堆在一个“数据管理”页面里。
2. 不允许跳过菜单拆分直接做质量、审计等后置能力。
3. 不允许在没有路网的项目中导入路段或病害。
4. 不允许把清除、删除作为普通用户主操作按钮突出展示。
5. 不允许新增一套与现有后台完全不同的视觉风格。
6. 不允许破坏旧路由兼容。
7. 不允许只做前端禁用而不规划后端校验。

---

## 7. 最小可交付版本

如果只能做最小版本，必须交付：

1. “数据管理”父菜单可展开/折叠。
2. 子菜单至少包含：
   - 项目总览
   - 项目导入
   - 导入记录
   - 模板与规范
3. `/admin/data-management` 重定向到项目总览。
4. 项目导入页保留现有上传能力。
5. 路段、病害导入入口明确提示依赖路网。
6. 前端构建通过。
